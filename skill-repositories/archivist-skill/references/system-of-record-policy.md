# System of Record Policy

A repository system of record preserves why the code changed, not only what changed.

## Required records

Every meaningful update should leave at least one durable markdown record:

- A `CHANGELOG.md` entry for release-facing summaries.
- A `docs/history` file for detailed maintainer context.
- Specialized notes for architecture decisions, migrations, APIs, releases, or deployments when relevant.

## What counts as meaningful

Create a record for:

- Features and user-visible changes.
- Bug fixes that required investigation.
- Refactors that alter boundaries or maintenance expectations.
- Dependency upgrades with compatibility or security impact.
- Public API, database, build, deployment, auth, or configuration changes.
- Security and privacy fixes.
- Releases, public artifacts, or production deployments.
- Changes generated by automation or AI that future maintainers may need to audit.

Small typo fixes, formatting-only commits, and comments-only changes can be recorded in the changelog only when the repository requires strict traceability.

## Record quality standard

A good record answers:

- What changed?
- Why did it change?
- Where did it change?
- Who or what made the change?
- How was it verified?
- What might break?
- How can it be rolled back?
- What should future maintainers watch next?

Use `Not recorded` when a field is unknown. Do not invent facts.

## Sensitive information rule

Records must never contain real secret values, access tokens, private keys, customer data, internal-only credentials, database passwords, or raw environment values.

It is safe to record variable names, example placeholders, and required configuration names.
